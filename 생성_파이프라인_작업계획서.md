# 생성 파이프라인 담당자 작업 정리

## 1. 역할 정의

생성 파이프라인 담당자는 사용자의 입력을 받아 아래 흐름을 안정적으로 완성하는 역할을 맡는다.

`입력(TEXT / SKETCH) -> AI 분석 -> Architecture JSON 생성 -> Terraform 코드 생성 -> Terraform 검증 -> 비용 추정 -> 세션 결과 저장`

현재 프로젝트 기준으로 주요 책임 모듈은 다음과 같다.

- `backend/app/ai_parser.py`
- `backend/app/terraform_generator.py`
- `backend/app/terraform_validator.py`
- `backend/app/cost_calculator.py`
- `backend/app/routers/sessions.py`
- `frontend/app/src/pages/Console/SketchConsole.tsx`

---

## 2. 현재 담당 범위

### 이미 구현된 범위

- 세션 단위 분석 API 존재
- 아키텍처 JSON 저장 API 존재
- Terraform 생성 및 validate API 존재
- 비용 계산 API 존재
- 프론트에서 생성 흐름 연결 존재
- 결과 상세 조회 및 세션 이력 조회 존재

### 담당자가 계속 책임져야 하는 범위

- 입력 해석 품질
- JSON 스키마 정합성
- Terraform 생성 품질
- Terraform 검증 성공률
- 비용 계산 정합성
- 파이프라인 API 계약 일관성
- 파이프라인 테스트 자동화

---

## 3. 우선순위별 해야 할 일

## 3-1. 1순위: Analyze API 계약 정리

목표:
생성 파이프라인 시작점의 요청/응답 규칙을 다른 API와 일관되게 맞춘다.

해야 할 일:

- `analyze` API의 snake_case 요청/응답 구조를 검토한다.
- 프로젝트 전체 API 규칙이 camelCase 중심이므로 `analyze` 응답도 같은 규칙으로 통일할지 결정한다.
- `contract_version`, `session_id`, `parsed_json` 같은 필드명을 전체 계약 기준으로 재정리한다.
- 프론트 `SketchConsole.tsx`와 백엔드 `schemas/session.py`를 동일 규격으로 맞춘다.
- 문서 `API_명세.md`와 실제 구현을 다시 맞춘다.

완료 기준:

- 생성 파이프라인 관련 API 필드명이 한 가지 규칙으로 통일된다.
- 프론트와 백엔드가 같은 계약을 기준으로 동작한다.
- 문서와 실제 응답 예시가 일치한다.

---

## 3-2. 2순위: 스키마 버전 저장 의미 정리

목표:
Architecture JSON 스키마 버전과 API 계약 버전을 구분한다.

현재 확인 사항:

- `sessions.py`의 analyze 저장 로직에서 `schema_version`에 `CONTRACT_VERSION(v2)`를 저장하고 있다.
- 하지만 `schema_version`은 JSON 구조 버전인 `v1` 성격에 가깝다.

해야 할 일:

- `schema_version` 필드의 의미를 명확히 정의한다.
- Architecture 저장 시 `schemaVersion`과 `contractVersion`이 혼동되지 않게 정리한다.
- `save_architecture`, `analyze`, `session detail` 응답에서 버전 값이 일관되게 보이도록 수정한다.
- 필요하면 문서에 버전 정책을 따로 추가한다.

완료 기준:

- DB에 저장되는 `schema_version` 값이 의미상 정확하다.
- API 계약 버전과 JSON 스키마 버전이 분리되어 설명된다.

---

## 3-3. 3순위: AI 입력 해석 품질 개선

목표:
TEXT / SKETCH 입력이 더 정확한 Architecture JSON으로 변환되도록 만든다.

현재 확인 사항:

- Bedrock 기반 파싱과 로컬 fallback이 구현되어 있다.
- 요구사항 반영률 계산, unmet hints, rationale도 구현되어 있다.
- 일부 한글 키워드/정규식은 인코딩 이슈 흔적이 있어 품질 점검이 필요하다.

해야 할 일:

- `ai_parser.py`의 한글 키워드, 정규식, fallback 규칙을 전수 점검한다.
- 입력 문장에서 EC2 수량, 인스턴스 타입, 리전, RDS 여부, Bedrock 여부, 추가 서비스 추출 정확도를 높인다.
- 이미지 입력 시 추론 실패 케이스를 수집하고 재현 가능한 테스트 입력을 만든다.
- `A_JSON_스키마_v1.json` 기준으로 누락/과잉 필드가 생기지 않게 보정 로직을 강화한다.
- 로컬 fallback 결과도 스키마를 안정적으로 만족하도록 유지한다.

완료 기준:

- 대표 시나리오에서 파싱 결과가 요구사항과 크게 어긋나지 않는다.
- fallback 사용 시에도 유효한 JSON이 안정적으로 생성된다.
- 입력 품질 저하 케이스를 재현하는 테스트 데이터가 존재한다.

---

## 3-4. 4순위: 이미지 업로드 실구현

목표:
현재 stub인 이미지 업로드를 실제 사용 가능한 업로드 흐름으로 바꾼다.

현재 확인 사항:

- `/api/uploads/images`는 현재 presigned URL 또는 실제 저장 없이 가짜 URL만 반환한다.

해야 할 일:

- 업로드 방식을 정한다.
- 선택지 1: 서버 직접 업로드
- 선택지 2: presigned URL 발급
- 이미지 파일 저장 위치와 접근 방식을 정한다.
- 업로드 결과 URL이 세션 생성과 analyze 흐름에서 실제로 사용되도록 연결한다.
- 보안 제한과 파일 타입 검증을 추가한다.

완료 기준:

- 업로드된 이미지가 실제 저장된다.
- 세션 생성 시 `inputImageUrl`이 실제 값으로 저장된다.
- 스케치 기반 analyze 흐름이 운영 가능한 수준으로 동작한다.

---

## 3-5. 5순위: Terraform 생성 범위와 품질 정리

목표:
생성 가능한 리소스와 미지원 리소스를 명확히 하고 Terraform 결과 품질을 높인다.

현재 확인 사항:

- EC2, RDS, Bedrock, S3, DynamoDB, SQS, SNS는 직접 생성한다.
- 그 외 추가 서비스는 `locals.additional_service_requirements`로만 남긴다.

해야 할 일:

- 지원 대상 서비스 목록을 문서로 고정한다.
- 미지원 서비스는 주석/메타정보/에러 중 어떤 방식으로 처리할지 정한다.
- public/private 여부에 따른 네트워크 구성이 현재처럼 단순화되어도 되는지 검토한다.
- 하드코딩된 값(AMI, DB 비밀번호, subnet 구조 등)을 MVP용/확장용으로 분리한다.
- Terraform 코드가 validate 가능한 최소 템플릿을 계속 유지한다.

완료 기준:

- 지원 서비스 범위가 문서화되어 있다.
- 미지원 서비스 처리 방식이 일관된다.
- 생성 결과가 예측 가능하고 validate 실패 원인이 줄어든다.

---

## 3-6. 6순위: Terraform 검증 실행환경 안정화

목표:
개발 환경마다 다른 Terraform 설치 상태 때문에 파이프라인이 흔들리지 않게 한다.

현재 확인 사항:

- `terraform_validator.py`는 로컬 Terraform 바이너리에 의존한다.

해야 할 일:

- 로컬 개발 환경, CI 환경, 배포 환경에서 Terraform 실행 경로를 정리한다.
- `terraform init`, `terraform validate` 실패 로그를 더 보기 쉽게 다듬는다.
- Terraform 미설치 환경에서의 대응 방식을 정한다.
- 필요하면 사전 점검 스크립트에 Terraform 존재 여부를 추가한다.

완료 기준:

- 주요 실행 환경에서 validate가 재현 가능하다.
- 실패 시 원인을 빠르게 파악할 수 있다.

---

## 3-7. 7순위: 비용 계산 정합성 유지

목표:
Architecture JSON과 Terraform 생성 방향에 맞는 비용 추정 결과를 제공한다.

현재 확인 사항:

- Pricing API 우선, 실패 시 정적 테이블 fallback 구조가 구현되어 있다.
- Bedrock 비용과 추가 서비스 usage 기반 비용도 계산한다.

해야 할 일:

- parser가 생성하는 필드와 cost calculator가 기대하는 필드가 항상 맞는지 점검한다.
- 추가 서비스별 비용 추정 로직의 가정을 문서화한다.
- `assumptionJson`의 의미와 UI 노출 기준을 정리한다.
- what-if 시나리오와 optimization 결과가 사용자에게 이해 가능한지 검토한다.

완료 기준:

- 비용 계산 입력 구조가 파이프라인 전체와 일치한다.
- 어떤 값이 추정치인지 사용자와 개발자가 모두 이해할 수 있다.

---

## 3-8. 8순위: 파이프라인 테스트 강화

목표:
생성 파이프라인의 핵심 흐름이 깨졌을 때 바로 발견되게 만든다.

현재 확인 사항:

- `backend/tests/test_sessions_pipeline.py`에 일부 테스트가 있다.
- 하지만 `backend/scripts/preflight_check.py`의 기본 unit_tests 대상에는 아직 포함되지 않았다.

해야 할 일:

- 프로젝트 생성 -> 세션 생성 -> analyze -> terraform -> cost -> detail 조회 전체 흐름 테스트를 보강한다.
- analyze 실패, schema 실패, terraform validate 실패, cost fallback 케이스를 테스트한다.
- `test_sessions_pipeline.py`를 preflight 실행 대상에 포함한다.
- 스케치 입력 케이스와 텍스트 입력 케이스를 나눠 검증한다.

완료 기준:

- 생성 파이프라인 핵심 흐름 테스트가 CI성 체크에 포함된다.
- 주요 실패 케이스가 자동으로 감지된다.

## 7. 최종 목표

생성 파이프라인 담당자의 최종 목표는 단순히 JSON이나 Terraform을 한 번 생성하는 것이 아니다.

아래 흐름이 항상 재현 가능하고 검증 가능하도록 만드는 것이 목표다.

`입력 -> 분석 -> 구조화 -> 코드 생성 -> 검증 -> 비용 추정 -> 결과 저장 -> 재조회`

이 흐름이 안정화되면 생성 결과가 일회성 데모가 아니라 실제 서비스 기능으로 동작하게 된다.
