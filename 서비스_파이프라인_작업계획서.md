# 서비스 파이프라인 작업계획서

## 1. 문서 목적

이 문서는 `작업계획서.md`의 B 역할인 서비스 파이프라인 담당자가
현재 저장소 상태를 기준으로 다음에 무엇을 해야 하는지 정리한 실행 문서다.

기존 계획서는 역할과 순서를 정의하는 문서이고,
이 문서는 실제 구현 현황을 반영한 후속 작업 체크리스트다.

핵심 목표는 다음과 같다.

- 생성 파이프라인 결과를 서비스 구조 안에서 안정적으로 저장/조회할 수 있게 유지한다.
- 프론트와 백엔드의 API 흐름을 하나로 정리한다.
- 세션 이력과 비교 기능을 사용 가능한 수준으로 연결한다.
- 빈 DB에서도 재현 가능한 마이그레이션과 테스트 체계를 맞춘다.

---

## 2. 현재 상태 요약

서비스 파이프라인 관점에서 이미 구현된 항목은 아래와 같다.

### 완료 또는 거의 완료된 항목

- `Project`, `Session`, `SessionArchitecture`, `SessionTerraformResult`, `SessionCostResult`, `SessionEvent` 모델 존재
- 프로젝트 생성 / 목록 조회 API 존재
- 프로젝트별 세션 생성 / 목록 조회 API 존재
- 세션 상세 조회 API 존재
- 아키텍처 저장 API 존재
- Terraform 생성 및 저장 API 존재
- 비용 계산 및 저장 API 존재
- 세션 상태 전이 로직 존재
- 세션 이벤트 기록 로직 존재
- 세션 비교 API 존재
- 인증 기반 owner 접근 제어 존재

### 이미 구현되어 있지만 정리가 필요한 항목

- 세션 이력/비교 기능은 백엔드 준비가 되어 있지만 프론트 연결은 아직 부족함
- 문서상 계약과 실제 프론트 UX 흐름을 더 촘촘히 맞출 필요가 있음
- DB 원본 스키마는 SQL에 잘 정리되어 있지만 Alembic 초기화 수준은 아직 부족함
- 테스트는 일부만 있고 서비스 파이프라인 전 구간 회귀 테스트는 부족함

### 현 시점 판단

서비스 파이프라인 담당자의 다음 단계는
"새 도메인 설계"가 아니라
"이미 만든 서비스 구조를 표준 흐름으로 정리하고 검증하는 작업"이다.

## 3. 우선순위 요약

지금부터의 작업 우선순위는 아래 순서를 권장한다.

1. API 단일화
2. 프론트 연결 정리
3. 이력/비교 기능 연결
4. DB 마이그레이션 정비
5. 테스트 보강
6. 운영성/문서 보완

이 순서를 추천하는 이유는,
현재 가장 큰 리스크가 "기능 부재"보다 "흐름 혼선"이기 때문이다.

## 4. 지금 당장 해야 할 일

## 4-1. 1순위: API 단일화

목표:
프론트와 백엔드가 같은 계약을 보도록 만든다.

배경:

- 백엔드에는 `/api/projects`, `/api/projects/{projectId}/sessions`, `/api/sessions/{sessionId}` 계열이 이미 준비되어 있다.
- 세션 관련 레거시 `/sessions/...` 엔드포인트는 정리되었고, 이제 `/api/...` 계약이 공식 기준이다.
- 이제 남은 과제는 이 공식 계약을 프론트 UX와 테스트에 충분히 반영하는 것이다.

작업:

- 서비스 파이프라인 기준 최종 표준 API를 확정한다.
- 가능하면 프론트는 전부 `/api/...` 기준으로 맞춘다.
- `API_명세.md`와 실제 FastAPI 라우터를 다시 한 번 대조한다.
- 응답 필드 네이밍(`camelCase`, `contractVersion`)이 일관적인지 확인한다.

완료 기준:

- 프론트 메인 생성 흐름이 최신 `/api/...` 계약만 사용한다.
- 문서와 실제 엔드포인트가 일치한다.
- 중복 엔드포인트의 역할이 정리된다.

---

## 4-2. 2순위: 프론트 생성 흐름 정리

목표:
서비스 파이프라인의 표준 흐름으로 프론트가 결과를 만들고 다시 조회하도록 맞춘다.

현재 흐름:

`프로젝트 생성 -> 세션 생성 -> analyze 호출 -> terraform 생성 -> cost 생성 -> 상세 조회`

문제:

- 생성 흐름 중 일부가 서비스 파이프라인 문서와 분리되어 있다.
- 세션을 만들고도 프로젝트/세션 목록이나 이력 UX로 이어지지 않는다.
- "현재 세션이 프로젝트 안에서 몇 번째 버전인지"가 UI에서 충분히 드러나지 않는다.

작업:

- 대시보드에서 생성 직후 `projectId`, `sessionId`, `versionNo`를 상태로 보관한다.
- 생성 후 결과만 보여주는 것이 아니라 현재 프로젝트 기준 세션 목록도 다시 불러온다.
- 세션 상세 조회를 기준으로 UI 상태를 복원하도록 정리한다.
- 프론트에서 세션 단건 응답을 source of truth로 사용하도록 통일한다.
- 실패 시 `requestId`, `error.code`, `error.message`를 더 잘 드러내도록 정리한다.

완료 기준:

- 새로고침 후에도 특정 세션 상세를 다시 조회해 화면을 복구할 수 있다.
- 세션 목록과 상세가 같은 데이터 구조를 기준으로 동작한다.
- 프론트가 "임시 결과 상태"보다 "백엔드 저장 결과"를 기준으로 화면을 그린다.

---

## 4-3. 3순위: 세션 이력 / 비교 기능 연결

목표:
작업계획서의 `B-5`를 실제 사용자 기능으로 최소 완성한다.

현재 상태:

- 백엔드에는 `version_no`가 있다.
- 프로젝트별 세션 목록 조회가 있다.
- 세션 비교 API(`/api/sessions/{sessionId}/compare`)도 있다.
- 하지만 프론트는 아직 비교 기능을 거의 쓰지 않는다.

작업:

- 프로젝트별 세션 목록 UI를 만든다.
- 각 세션의 `versionNo`, `status`, `createdAt`을 보이게 한다.
- 특정 세션 선택 시 상세 조회를 다시 호출하는 구조를 만든다.
- 이전 버전과 비교하는 버튼을 붙인다.
- 비교 응답의 `jsonDiff`, `terraformDiff`, `costDiff`를 읽기 쉬운 UI로 보여준다.

최소 구현 기준:

- 세션 목록에서 이전 버전을 선택할 수 있다.
- 최신 세션과 직전 세션 비교가 가능하다.
- JSON 변경점, Terraform diff, 비용 차이를 확인할 수 있다.

후순위:

- diff 시각화 고도화
- 변화 요약 AI 설명
- 비교 결과 다운로드

---

## 4-4. 4순위: DB 마이그레이션 정비

목표:
현재 SQL 원본 스키마와 Alembic 기반 실제 초기화 절차를 맞춘다.

배경:

- `backend/db/schema_v1.sql`은 서비스 파이프라인 구조를 잘 반영하고 있다.
- 하지만 Alembic은 부분 보정 성격이 강하고, 새 환경에서 DB를 완전히 재현하는 기준으로는 약하다.

작업:

- Alembic 기준으로도 `users`, `projects`, `sessions`, `session_architectures`, `session_terraform_results`, `session_cost_results`, `session_events`가 모두 재현되는지 점검한다.
- 신규 DB에서 `alembic upgrade head`만으로 부팅 가능한지 확인한다.
- 필요하면 초기 스키마 생성용 migration을 보강한다.
- SQL 원본과 ORM 모델과 Alembic 결과가 서로 어긋나지 않게 맞춘다.

완료 기준:

- 새 DB에서 migration만으로 서비스 실행 가능
- 모델, SQL 문서, migration 결과가 일치
- 로컬 개발자 온보딩 시 DB 초기화 절차가 단순해짐

---

## 4-5. 5순위: 테스트 보강

목표:
서비스 파이프라인 전체 흐름이 깨졌을 때 바로 알 수 있게 만든다.

현재 상태:

- 비용 결과 일부에 대한 테스트는 있다.
- 하지만 CRUD, 상태 전이, 권한, compare, 실패 케이스 테스트는 부족하다.

우선 추가할 테스트:

- 프로젝트 생성 / 목록 조회 성공 케이스
- 다른 사용자 프로젝트 접근 차단 케이스
- 세션 생성 시 `versionNo` 증가 케이스
- 세션 상세 조회 응답 구조 검증
- 아키텍처 저장 후 상태가 `ANALYZED`로 바뀌는지 검증
- Terraform 생성 후 상태와 validation 결과 검증
- 비용 계산 후 상태와 assumption 저장 검증
- compare API가 직전 세션 기준 diff를 반환하는지 검증
- 잘못된 상태 전이 요청이 `409`를 반환하는지 검증

완료 기준:

- 서비스 파이프라인 핵심 API에 대해 회귀 테스트가 존재
- 세션 버전과 비교 기능이 테스트로 보호됨
- 권한 체크가 테스트로 보장됨

---

## 4-6. 6순위: 운영성 / 문서 보완

목표:
나중에 팀원이 바뀌어도 서비스 파이프라인 구조를 바로 이해할 수 있게 만든다.

작업:

- `API_명세.md`를 실제 구현 기준으로 다시 정리
- 서비스 파이프라인 기준 시퀀스 문서 작성
- 상태 전이 규칙 문서화
- 실패 응답 규칙 문서화
- 세션 비교 기능 사용법 문서화

권장 추가 문서:

- `backend/docs/서비스_세션_흐름.md`
- `backend/docs/상태전이_규칙.md`
- `backend/docs/세션비교_API.md`

## 5. 작업계획서 기준 매핑

기존 `작업계획서.md`의 B 단계와 현재 상태를 매핑하면 아래와 같다.

### Step B-1. 프로젝트/세션 구조 설계

상태:
- 완료

근거:
- DB 스키마, ORM 모델, API 구조가 이미 잡혀 있음

남은 일:
- 문서와 migration만 더 정리

### Step B-2. 결과 저장 및 조회 API 구현

상태:
- 대부분 완료

근거:
- 프로젝트 생성/조회
- 세션 생성/조회
- 아키텍처/Terraform/비용 저장 및 조회 가능

남은 일:
- 구형 엔드포인트 정리
- 프론트에서 최신 계약만 사용하도록 통일

### Step B-3. 상태 관리 흐름 구현

상태:
- 완료에 가까움

근거:
- 상태 전이 규칙 존재
- 에러 코드/메시지 저장 존재
- 이벤트 로그 기록 존재

남은 일:
- 예외 케이스 테스트 보강
- 프론트 표시 UX 보완

### Step B-4. 비용 계산 로직 구현

상태:
- 완료

근거:
- 비용 계산 모듈 존재
- 가정값 저장 존재
- 최적화 추천과 시나리오까지 구현돼 있음

남은 일:
- 테스트와 문서 정리

### Step B-5. 이력 관리 구조 설계

상태:
- 구조는 완료, 사용자 기능 연결은 미완료

근거:
- `version_no`, 세션 목록, 비교 API 존재

남은 일:
- 프론트에서 세션 이력/비교 UI 연결

## 8. 최종 목표

서비스 파이프라인 담당자의 최종 목표는
"생성 결과를 보여주는 백엔드"를 만드는 것이 아니다.

진짜 목표는 아래 흐름을 서비스처럼 자연스럽게 만드는 것이다.

`프로젝트 생성 -> 세션 생성 -> 결과 저장 -> 세션 조회 -> 버전 누적 -> 이전 결과와 비교`

이 흐름이 안정적으로 돌아가면,
생성 파이프라인 담당자가 만든 결과가 실제 서비스 경험으로 연결된다.

즉 지금부터의 핵심은 새 기능을 많이 추가하는 것보다,
이미 만든 구조를 하나의 일관된 서비스 흐름으로 완성하는 것이다.
