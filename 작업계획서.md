# Sketch-to-Cloud 작업계획서

## 1. 문서 목적

이 문서는 `README.md`의 팀 구성 및 역할 분담을 기준으로,
팀원 A와 팀원 B가 실제로 어떤 작업을 어떤 순서로 진행해야 하는지 정리한 실행 문서이다.

기준 원칙은 다음과 같다.

- 프론트/백 분리가 아니라 기능 파이프라인 기준으로 작업한다.
- 4주 안에 `입력 -> JSON -> Terraform -> 비용 -> 시각화` 흐름을 완성한다.
- MVP 범위는 `VPC`, `EC2`, `RDS`로 고정한다.
- 입력은 `텍스트 우선`, 스케치는 보조 기능으로 처리한다.

---

## 2. 역할 정의

### A. 생성 파이프라인 담당

A는 사용자의 입력을 실제 인프라 설계 결과로 바꾸는 파이프라인을 담당한다.

주요 책임:

- 입력 해석 방식 정의
- AI 프롬프트 설계
- 아키텍처 JSON 스키마 설계
- JSON 생성 로직 구현
- Terraform 코드 생성 로직 구현
- 생성 결과의 유효성 검증
- 보안 체크 로직의 초안 설계

### B. 서비스 파이프라인 담당

B는 생성 결과를 저장하고 보여주며, 서비스처럼 동작하게 만드는 부분을 담당한다.

주요 책임:

- 프로젝트/세션 구조 설계
- 결과 저장 및 조회 API 구현
- 비용 계산 로직 구현
- 상태 관리 방식 설계
- 생성 결과 이력 관리
- 프론트와 백엔드 연결을 위한 API 흐름 정리

### 공통 책임

- API 명세 확정
- React / vite 프론트엔드 UI 구현
- 통합 테스트
- 데모 시나리오 작성
- 문서화 및 산출물 정리

---

## 3. 전체 작업 순서

### 1단계. 공통 준비

이 단계는 A와 B가 같이 맞춰야 한다.

- 프로젝트 목표와 MVP 범위를 다시 확정한다.
- 지원 리소스를 `VPC`, `EC2`, `RDS`로 제한한다.
- 입력 형식을 `텍스트 우선`, `스케치 보조`로 확정한다.
- 최종 출력 형식을 `JSON`, `Terraform 코드`, `비용 결과`, `시각화 데이터`로 정한다.
- API 요청/응답 형식을 먼저 합의한다.

이 단계가 끝나야 각자 작업을 병렬로 진행할 수 있다.

## 4. A 작업 순서

### Step A-1. 입력 시나리오 수집

목표:
실제로 어떤 요구사항이 들어올지 기준 데이터를 만든다.

작업:

- 대표 입력 예시 20개 작성
- 쉬운 케이스와 애매한 케이스를 섞어서 준비
- 예시마다 기대 결과를 간단히 적어둠
- 텍스트 입력과 스케치 입력을 분리해서 관리

산출물:

- 테스트 입력 목록
- 입력별 기대 JSON 초안

### Step A-2. JSON 스키마 정의

목표:
AI가 만들어낼 중간 결과 형식을 고정한다.

작업:

- `vpc`, `ec2`, `rds`, `public` 같은 핵심 필드 정의
- 필수값과 선택값 구분
- 잘못된 입력이 들어왔을 때 비어도 되는 필드 정의
- Terraform 생성에 필요한 최소 정보만 포함

산출물:

- JSON 스키마 v1
- 입력 예시별 JSON 예시

### Step A-3. 프롬프트 설계 및 AI 해석 로직 구현

목표:
입력을 안정적으로 JSON으로 변환한다.

작업:

- Bedrock 호출 방식 결정
- 시스템 프롬프트와 사용자 프롬프트 작성
- 출력 형식을 JSON으로 강제
- 예외 응답, 누락 응답, 이상한 응답에 대한 재시도 전략 정의
- 텍스트 입력 먼저 붙이고, 이후 스케치 입력 연결

산출물:

- 프롬프트 초안
- JSON 생성 API 또는 모듈
- 샘플 입력 20개 기준 테스트 결과

검증 기준:

- 해석 정확도 70% 이상
- 주요 필드 누락률 낮을 것
- 애매한 입력에 대해 최소한 일관된 구조를 반환할 것

### Step A-4. Terraform 생성 로직 구현

목표:
JSON 결과를 Terraform 코드로 변환한다.

작업:

- `VPC`, `EC2`, `RDS` 템플릿 설계
- JSON 값에 따라 템플릿 분기 처리
- 리소스 간 연결 관계 반영
- 생성된 코드가 너무 복잡해지지 않도록 옵션 제한
- `terraform validate` 기준으로 유효성 검증

산출물:

- JSON -> Terraform 생성기
- 템플릿 파일 또는 매핑 규칙
- validate 테스트 결과

검증 기준:

- validate 통과율 80% 이상
- 리소스 연결 구조가 깨지지 않을 것
- 입력별로 코드 형태가 지나치게 흔들리지 않을 것

### Step A-5. 보안 체크 초안

목표:
Phase 2로 넘기더라도 최소한의 구조는 잡아둔다.

작업:

- 퍼블릭 노출 여부 체크 기준 정의
- 열린 포트 경고 기준 정의
- Terraform 생성 결과에서 체크 가능한 항목만 우선 추출

산출물:

- 보안 체크 룰 목록 초안

비고:

- 일정이 밀리면 이 단계는 구현보다 문서화만 하고 Phase 2로 넘긴다.

---

## 5. B 작업 순서

### Step B-1. 프로젝트/세션 구조 설계

목표:
사용자 입력과 생성 결과를 저장할 기본 단위를 만든다.

작업:

- 프로젝트 엔티티 정의
- 세션 엔티티 정의
- 입력 원문, JSON 결과, Terraform 결과, 비용 결과 저장 구조 설계
- 상태값 정의 (`created`, `analyzing`, `generated`, `failed` 등)

산출물:

- DB 스키마 초안
- 상태 전이 규칙

### Step B-2. 결과 저장 및 조회 API 구현

목표:
프론트에서 생성 결과를 저장하고 다시 조회할 수 있게 한다.

작업:

- 프로젝트 생성 API
- 세션 생성 API
- 분석 결과 저장 API
- Terraform 결과 저장 API
- 비용 결과 저장 API
- 단건 조회 및 목록 조회 API

산출물:

- API 명세 초안
- 기본 CRUD 구현

### Step B-3. 상태 관리 흐름 구현

목표:
현재 작업이 어디까지 진행됐는지 서비스 단에서 추적 가능하게 한다.

작업:

- 입력 후 상태 변경 흐름 정의
- A의 생성 파이프라인 완료 시점과 연결
- 실패 시 에러 메시지 저장 규칙 정의
- 재시도 가능한 상태와 불가능한 상태 구분

산출물:

- 상태 전이 로직
- 에러 처리 규칙

### Step B-4. 비용 계산 로직 구현

목표:
생성된 리소스를 기준으로 월 예상 비용을 계산한다.

작업:

- 비용 계산 기준값 정의
- 리전, 인스턴스 타입, 사용시간 등 가정값 테이블 정의
- `EC2`, `RDS` 중심 계산 로직 구현
- 총합 계산 및 상세 항목 분리
- 결과에 가정값을 함께 저장

산출물:

- 비용 계산 모듈
- 비용 응답 포맷

검증 기준:

- 모든 비용 응답에 가정값 포함
- 계산 실패 시 원인 메시지 명시
- 복잡한 경우 범위값으로 대체 가능

### Step B-5. 이력 관리 구조 설계

목표:
생성 결과를 버전처럼 비교할 수 있는 기반을 만든다.

작업:

- 세션별 버전 번호 규칙 정의
- 이전 결과와 현재 결과 비교 기준 정의
- JSON, Terraform, 비용 결과를 묶어서 저장

산출물:

- 이력 관리 초안

비고:

- 일정이 부족하면 조회만 구현하고 비교 기능은 Phase 2로 넘긴다.

## 6. 공통 작업 순서

### Step C-1. API 명세 고정

시점:
A의 JSON 스키마 초안과 B의 세션 구조 초안이 나온 직후

작업:

- 입력 API 형식 확정
- FastAPI 기준 생성 결과 응답 형식 확정
- 에러 응답 형식 확정
- React / vite 프론트가 바로 붙일 수 있게 예시 payload 작성

### Step C-2. 프론트 연결

시점:
A의 JSON/Terraform 결과와 B의 저장 API가 준비된 뒤

작업:

- 입력 폼 구현
- 결과 화면 구성
- React Flow 다이어그램 연결
- Terraform 코드 출력 영역 연결
- 비용 요약 UI 연결
- FastAPI API 연동 방식 정리

### Step C-3. 통합 테스트

시점:
주요 기능이 모두 한 번씩 연결된 뒤

작업:

- 데모 시나리오 3개 선정
- 정상 케이스 테스트
- 실패 케이스 테스트
- 폴백 동작 확인
- API와 프론트 연결 에러 수정

### Step C-4. 데모/문서화

시점:
4주차 마지막

작업:

- 발표용 데모 입력 3개 확정
- README 최종 정리
- 아키텍처 흐름도 작성
- 역할별 기여 내용 정리

## 7. 주차별 권장 배치

### 1주차

- A: 입력 시나리오 수집, JSON 스키마 정의, 프롬프트 초안 작성
- B: 프로젝트/세션 구조 설계, DB 초안, 상태값 정의
- 공통: API 명세 1차 합의

### 2주차

- A: AI 해석 로직 구현, JSON 생성 안정화
- B: 저장/조회 API 구현, 상태 전이 로직 연결
- 공통: JSON 결과를 프론트에서 임시로 확인

### 3주차

- A: Terraform 생성기 구현, validate 검증
- B: 비용 계산 로직 구현
- 공통: React Flow, 코드 출력, 비용 화면 연결

### 4주차

- A: 해석 실패/생성 실패 케이스 정리
- B: 이력 관리 최소 구현, 에러 처리 정리
- 공통: 통합 테스트, 데모 시나리오 확정, 발표 문서 정리

## 8. 체크포인트

### 체크포인트 1. 1주차 종료 시점

- JSON 스키마가 확정되었는가
- 프로젝트/세션 구조가 확정되었는가
- A와 B가 같은 데이터 형식을 보고 있는가

### 체크포인트 2. 2주차 종료 시점

- 텍스트 입력 기준 JSON 생성이 되는가
- 결과 저장/조회 API가 동작하는가
- 실패 응답 형식이 정리되었는가

### 체크포인트 3. 3주차 종료 시점

- JSON -> Terraform 변환이 되는가
- 비용 계산이 최소 수준이라도 표시되는가
- 프론트에서 결과 3종(JSON, 코드, 비용)을 모두 볼 수 있는가

### 체크포인트 4. 4주차 종료 시점

- 데모 시나리오 3개 중 2개 이상 성공하는가
- 실패 케이스에서 폴백이 동작하는가
- 발표 가능한 수준으로 흐름 설명이 가능한가

## 9. 우선순위 정리

시간이 부족할 때도 반드시 끝내야 하는 항목:

- 텍스트 입력 -> JSON 생성
- JSON -> Terraform 생성
- 비용 계산 최소 버전
- 결과 시각화
- 저장/조회 API

시간이 부족하면 뒤로 미뤄도 되는 항목:

- 스케치 입력 고도화
- 보안 체크 구현
- 이력 비교 상세 기능
- AI 채팅
- LocalStack 시뮬레이션 고도화

## 10. 최종 목표

팀원 A와 B는 각자 기능을 따로 만드는 것이 아니라,
마지막에는 하나의 흐름으로 연결되는 결과를 만들어야 한다.

최종적으로 보여줘야 하는 것은 다음 한 줄이다.

`사용자 입력 -> AI 해석 -> Terraform 생성 -> 비용 계산 -> 시각화`

이 흐름이 데모에서 자연스럽게 동작하면 이번 작업은 성공이다.
