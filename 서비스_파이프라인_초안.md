# 서비스 파이프라인 초안

## 1. 문서 목적

이 문서는 `Sketch-to-Cloud` 프로젝트에서 서비스 파이프라인 담당자가
바로 구현을 시작할 수 있도록 도메인 구조, 상태 흐름, API, 구현 우선순위를 정리한 초안이다.

기준 문서는 다음과 같다.

- `README.md`
- `작업계획서.md`

본 문서는 특히 아래 역할을 구체화한다.

- 프로젝트/세션 구조 설계
- 결과 저장 및 조회 API 설계
- 상태 관리 설계
- 비용 계산 결과 저장 구조 설계
- 이력 관리의 최소 기반 설계
- React / vite 프론트엔드와 Python (FastAPI) 백엔드 연결 기준 정리

---

## 2. 설계 원칙

- MVP 범위는 `VPC`, `EC2`, `RDS`로 제한한다.
- 입력은 `텍스트 우선`, `스케치 보조`로 처리한다.
- 서비스 파이프라인은 생성 파이프라인의 결과를 저장하고 조회 가능하게 만드는 데 집중한다.
- 결과는 세션 단위로 관리한다.
- 전용 플랫폼 기준으로 모든 프로젝트와 세션은 로그인한 사용자 계정에 귀속된다.
- 이력 관리를 고려해 재시도는 기존 세션 수정 대신 새 세션 생성 방식으로 처리한다.

---

## 3. 핵심 도메인 구조

서비스 파이프라인의 기본 구조는 `Project -> Session` 2단계로 잡는다.

플랫폼 관점에서는 이 구조 앞단에 `User`가 존재하며,
실제 관계는 `User -> Project -> Session`으로 이해하는 것이 맞다.

### 3-0. User

로그인 가능한 플랫폼 사용자다.

주요 역할:

- 프로젝트와 세션의 소유 주체가 된다.
- 이메일/비밀번호 또는 소셜 로그인 계정을 식별한다.
- 권한, 접근 제어, 마지막 로그인 시각의 기준이 된다.

비고:

- 관리자 여부는 로그인 세션 테이블이 아니라 사용자 권한 필드 또는 별도 역할 테이블로 구분한다.

### 3-1. Project

사용자의 큰 작업 단위다.  
(예: "쇼핑몰 인프라 설계", "블로그 서비스 구성")

주요 역할:

- 여러 생성 시도(Session)를 묶는다.
- 사용자의 작업 컨텍스트를 유지한다.
- 세션 목록과 버전 흐름의 기준점이 된다.

### 3-2. Session

한 번의 생성 시도 또는 한 버전의 결과물이다.

주요 역할:

- 입력 원문 보관
- 생성 상태 추적
- JSON 결과 저장
- Terraform 결과 저장
- 비용 결과 저장
- 실패 정보 저장

권장 원칙:

- 세션 하나가 결과물 한 세트를 대표한다.
- 재시도 시 기존 세션을 덮어쓰지 않고 새 세션을 만든다.

## 4. 테이블 초안

## 4-1. users

이 테이블은 플랫폼 사용자 계정의 기본 정보를 저장한다.
로그인 주체, 프로젝트 소유자, 권한 구분의 기준이 된다.

| 필드명          | 타입              | 설명                             |
| --------------- | ----------------- | -------------------------------- |
| `id`            | UUID PK           | 사용자 ID                        |
| `email`         | VARCHAR(255)      | 로그인 이메일, UNIQUE            |
| `password_hash` | VARCHAR(255) NULL | 로컬 로그인용 비밀번호 해시      |
| `display_name`  | VARCHAR(100)      | 화면 표시 이름                   |
| `is_active`     | BOOLEAN           | 계정 활성 여부                   |
| `role`          | VARCHAR(20)       | 사용자 역할, 예: `ADMIN`, `USER` |
| `last_login_at` | TIMESTAMP NULL    | 마지막 로그인 시각               |
| `created_at`    | TIMESTAMP         | 생성 시각                        |
| `updated_at`    | TIMESTAMP         | 수정 시각                        |

비고:

- 로컬 로그인만 먼저 구현하더라도 이후 소셜 로그인을 고려해 사용자 기본 테이블은 분리해두는 것이 좋다.
- `password_hash`는 평문 비밀번호를 저장하지 않고 bcrypt/argon2 계열 해시만 저장한다.
- 계정 차단/블랙리스트 이력 관리는 현재 범위에서 제외하고, MVP에서는 `is_active`로 최소한의 활성 상태만 관리한다.
- 권한 구조가 단순한 MVP에서는 `users.role` 하나로 관리자/일반 사용자 구분을 처리하는 편이 단순하다.

## 4-2. auth_identities

이 테이블은 사용자의 로그인 수단을 관리한다.
로컬 로그인과 향후 소셜 로그인을 같은 구조 안에서 확장 가능하게 만든다.

| 필드명             | 타입         | 설명                                       |
| ------------------ | ------------ | ------------------------------------------ |
| `id`               | UUID PK      | 인증 식별자 ID                             |
| `user_id`          | UUID FK      | 연결된 사용자 ID                           |
| `provider`         | VARCHAR(30)  | `LOCAL`, `GOOGLE`, `GITHUB` 등             |
| `provider_user_id` | VARCHAR(255) | 외부 제공자 사용자 식별자, LOCAL이면 email |
| `created_at`       | TIMESTAMP    | 생성 시각                                  |
| `updated_at`       | TIMESTAMP    | 수정 시각                                  |

권장 제약:

- (`provider`, `provider_user_id`) UNIQUE
- 한 사용자는 여러 로그인 방식을 가질 수 있다.

## 4-3. auth_sessions

이 테이블은 로그인된 사용자의 세션을 관리한다.
로그인 유지, refresh token 관리, 로그아웃 처리의 기준이 된다.

| 필드명               | 타입              | 설명                              |
| -------------------- | ----------------- | --------------------------------- |
| `id`                 | UUID PK           | 로그인 세션 ID                    |
| `user_id`            | UUID FK           | 로그인한 사용자 ID                |
| `refresh_token_hash` | VARCHAR(255)      | refresh token 해시값              |
| `user_agent`         | VARCHAR(255) NULL | 접속 브라우저/앱 정보             |
| `ip_address`         | VARCHAR(64) NULL  | 접속 IP                           |
| `expires_at`         | TIMESTAMP         | 세션 만료 시각                    |
| `revoked_at`         | TIMESTAMP NULL    | 로그아웃 또는 강제 만료 처리 시각 |
| `created_at`         | TIMESTAMP         | 생성 시각                         |
| `updated_at`         | TIMESTAMP         | 수정 시각                         |

비고:

- Access token은 보통 무상태 JWT로 처리하고, DB에는 refresh token 계열만 저장하는 방식이 단순하다.
- 여러 디바이스 동시 로그인, 로그아웃, 세션 강제 만료를 지원하려면 별도 세션 테이블이 필요하다.
- 이 테이블은 관리자 전용이 아니라 일반 사용자와 관리자 모두가 공통으로 사용하는 로그인 세션 테이블이다.
- 관리자 권한 자체는 현재 문서 기준으로 `users.role` 필드에서 관리한다.

## 4-4. projects

이 테이블은 사용자의 큰 작업 단위인 프로젝트를 저장한다.
여러 세션을 묶는 상위 컨테이너 역할을 한다.

| 필드명        | 타입         | 설명                            |
| ------------- | ------------ | ------------------------------- |
| `id`          | UUID PK      | 프로젝트 ID                     |
| `name`        | VARCHAR(100) | 프로젝트 이름                   |
| `description` | TEXT NULL    | 프로젝트 설명                   |
| `owner_id`    | UUID FK NULL | 소유 사용자 ID, `users.id` 참조(인증 연동 전까지 nullable) |
| `created_at`  | TIMESTAMP    | 생성 시각                       |
| `updated_at`  | TIMESTAMP    | 수정 시각                       |

비고:

- 인증/인가 파이프라인이 완전히 연결되기 전에는 `owner_id`를 임시로 nullable 운영할 수 있다.
- 인증 연동 완료 시점에 `owner_id`를 `NOT NULL`로 전환하는 마이그레이션을 적용한다.
- ORM(예: SQLAlchemy)에서도 `projects.owner_id -> users.id` FK 해석을 위해 `users` 모델을 반드시 선언해야 한다.

## 4-5. sessions

세션은 서비스 파이프라인의 중심 테이블이며,
한 번의 생성 시도에 대한 입력, 상태, 에러 정보를 관리한다.

| 필드명            | 타입             | 설명                                 |
| ----------------- | ---------------- | ------------------------------------ |
| `id`              | UUID PK          | 세션 ID                              |
| `project_id`      | UUID FK          | 소속 프로젝트                        |
| `version_no`      | INT              | 프로젝트 내 버전 번호                |
| `input_type`      | VARCHAR(20)      | `TEXT`, `SKETCH`, `TEXT_WITH_SKETCH` |
| `input_text`      | TEXT NULL        | 텍스트 입력 원문                     |
| `input_image_url` | TEXT NULL        | 스케치 이미지 경로 또는 URL          |
| `status`          | VARCHAR(30)      | 현재 처리 상태                       |
| `error_code`      | VARCHAR(50) NULL | 실패 코드                            |
| `error_message`   | TEXT NULL        | 실패 메시지                          |
| `created_at`      | TIMESTAMP        | 생성 시각                            |
| `updated_at`      | TIMESTAMP        | 수정 시각                            |

비고:

- `sessions`는 서비스 파이프라인의 중심 테이블이다.
- 상태 관리와 조회 API 대부분이 이 테이블을 기준으로 움직인다.

## 4-6. session_architectures

이 테이블은 AI가 생성한 아키텍처 JSON 결과를 저장한다.
즉 입력 해석 단계의 구조화 결과를 보관하는 역할이다.

| 필드명              | 타입           | 설명                      |
| ------------------- | -------------- | ------------------------- |
| `id`                | UUID PK        | 결과 ID                   |
| `session_id`        | UUID FK UNIQUE | 세션과 1:1 연결           |
| `schema_version`    | VARCHAR(20)    | JSON 스키마 버전          |
| `architecture_json` | JSON 또는 TEXT | AI가 생성한 아키텍처 JSON |
| `created_at`        | TIMESTAMP      | 생성 시각                 |

## 4-7. session_terraform_results

이 테이블은 아키텍처 JSON을 기반으로 생성된 Terraform 코드와
유효성 검증 결과를 저장한다.

Terraform 코드는 사용자에게 전달되는 최종 산출물 성격이 강하다.
JSON이 내부 구조화 결과라면, Terraform은 사람이 읽고 수정하고 재사용할 수 있는 IaC(Infrastructure as Code, 인프라 구성을 콘솔 수작업 대신 코드로 정의하고 관리하는 방식) 결과물이다.

| 필드명              | 타입           | 설명                          |
| ------------------- | -------------- | ----------------------------- |
| `id`                | UUID PK        | 결과 ID                       |
| `session_id`        | UUID FK UNIQUE | 세션과 1:1 연결               |
| `terraform_code`    | TEXT           | 생성된 Terraform 코드         |
| `validation_status` | VARCHAR(20)    | `PENDING`, `PASSED`, `FAILED` |
| `validation_output` | TEXT NULL      | validate 결과 로그            |
| `created_at`        | TIMESTAMP      | 생성 시각                     |

## 4-8. session_cost_results

이 테이블은 생성된 인프라 기준의 예상 비용 결과를 저장한다.
총 비용, 상세 비용, 계산 가정값을 함께 보관한다.

| 필드명                | 타입           | 설명                 |
| --------------------- | -------------- | -------------------- |
| `id`                  | UUID PK        | 결과 ID              |
| `session_id`          | UUID FK UNIQUE | 세션과 1:1 연결      |
| `currency`            | VARCHAR(10)    | 예: `KRW`, `USD`     |
| `region`              | VARCHAR(30)    | 예: `ap-northeast-2` |
| `assumption_json`     | JSON 또는 TEXT | 비용 계산 가정값     |
| `monthly_total`       | DECIMAL(12,2)  | 월 예상 총 비용      |
| `cost_breakdown_json` | JSON 또는 TEXT | 비용 상세 항목       |
| `created_at`          | TIMESTAMP      | 생성 시각            |

## 5. 상태값 설계

세션 상태는 최소한 아래 단계로 관리한다.

| 상태값                 | 설명                     |
| ---------------------- | ------------------------ |
| `CREATED`              | 세션이 생성됨            |
| `ANALYZING`            | 입력 해석 중             |
| `ANALYZED`             | 아키텍처 JSON 생성 완료  |
| `GENERATING_TERRAFORM` | Terraform 코드 생성 중   |
| `GENERATED`            | Terraform 코드 생성 완료 |
| `COST_CALCULATED`      | 비용 계산 완료           |
| `FAILED`               | 처리 실패                |

### 상태 전이 규칙

- `CREATED` -> `ANALYZING`
- `ANALYZING` -> `ANALYZED`
- `ANALYZED` -> `GENERATING_TERRAFORM`
- `GENERATING_TERRAFORM` -> `GENERATED`
- `GENERATED` -> `COST_CALCULATED`
- 각 단계에서 실패 시 -> `FAILED`

### 실패 처리 규칙

- 실패 시 `error_code`, `error_message`를 남긴다.
- 세션 복구보다는 새 세션 생성으로 재시도한다.
- 재시도 시 `version_no`를 1 증가시킨다.

이 방식의 장점:

- 이력 관리가 쉬움
- 이전 결과와 현재 결과 비교가 쉬움
- 실패 원인 추적이 쉬움

## 6. 저장 단위와 조회 방식

서비스 파이프라인은 결과를 세션 단위로 저장한다.

세션 상세 조회 시 프론트가 한 번에 필요한 데이터는 다음과 같다.

- 입력 정보
- 현재 상태
- 아키텍처 JSON
- Terraform 코드
- 비용 결과
- 에러 정보

따라서 세션 상세 API는 여러 테이블을 묶어 한 번에 응답하는 형태가 적절하다.

또한 이 응답은 React / vite 프론트엔드가 React Flow 시각화, Terraform 코드 출력,
비용 요약 UI를 한 번에 구성할 수 있도록 설계하는 것이 좋다.

특히 Terraform 코드는 단순 참고용 텍스트가 아니라,
사용자가 인프라 정의 결과를 직접 검토하고 이후 배포 코드의 출발점으로 활용할 수 있는 핵심 결과물로 본다.

## 7. API 초안

## 7-1. 프로젝트 생성

`POST /api/projects`

요청 예시:

```json
{
  "name": "쇼핑몰 인프라 설계",
  "description": "EC2 2대와 RDS 포함"
}
```

응답 예시:

```json
{
  "projectId": "uuid",
  "name": "쇼핑몰 인프라 설계"
}
```

## 7-2. 프로젝트 목록 조회

`GET /api/projects`

## 7-3. 세션 생성

`POST /api/projects/{projectId}/sessions`

요청 예시:

```json
{
  "inputType": "TEXT",
  "inputText": "EC2 2개와 MySQL RDS 1개를 퍼블릭하게 구성"
}
```

응답 예시:

```json
{
  "sessionId": "uuid",
  "versionNo": 1,
  "status": "CREATED"
}
```

## 7-4. 아키텍처 JSON 저장

`POST /api/sessions/{sessionId}/architecture`

요청 예시:

```json
{
  "schemaVersion": "v1",
  "architectureJson": {
    "vpc": true,
    "ec2": 2,
    "rds": "mysql",
    "public": true
  }
}
```

## 7-5. Terraform 결과 저장

`POST /api/sessions/{sessionId}/terraform`

요청 예시:

```json
{
  "terraformCode": "resource ...",
  "validationStatus": "PASSED",
  "validationOutput": ""
}
```

## 7-6. 비용 결과 저장

`POST /api/sessions/{sessionId}/cost`

요청 예시:

```json
{
  "currency": "KRW",
  "region": "ap-northeast-2",
  "assumptionJson": {
    "ec2Type": "t3.micro",
    "rdsType": "db.t3.micro",
    "hoursPerMonth": 730
  },
  "monthlyTotal": 30000,
  "costBreakdownJson": {
    "ec2": 12000,
    "rds": 18000
  }
}
```

## 7-7. 세션 단건 조회

`GET /api/sessions/{sessionId}`

응답 예시:

```json
{
  "sessionId": "uuid",
  "projectId": "uuid",
  "versionNo": 1,
  "inputType": "TEXT",
  "inputText": "EC2 2개와 MySQL RDS 1개를 퍼블릭하게 구성",
  "status": "COST_CALCULATED",
  "architecture": {
    "schemaVersion": "v1",
    "architectureJson": {
      "vpc": true,
      "ec2": 2,
      "rds": "mysql",
      "public": true
    }
  },
  "terraform": {
    "validationStatus": "PASSED",
    "terraformCode": "resource ..."
  },
  "cost": {
    "currency": "KRW",
    "region": "ap-northeast-2",
    "monthlyTotal": 30000,
    "costBreakdownJson": {
      "ec2": 12000,
      "rds": 18000
    }
  },
  "error": null
}
```

## 7-8. 프로젝트별 세션 목록 조회

`GET /api/projects/{projectId}/sessions`

## 7-9. 상태 변경

`PATCH /api/sessions/{sessionId}/status`

요청 예시:

```json
{
  "status": "ANALYZING",
  "errorCode": null,
  "errorMessage": null
}
```

비고:

- 실제 구현에서는 상태 변경 API를 외부에 직접 열지 않고 내부 서비스 로직으로만 처리해도 된다.
- 다만 React / vite 프론트엔드와 Python (FastAPI) 백엔드 역할 분리가 크지 않은 팀 구조라면 임시 API로 두고 개발 속도를 높일 수 있다.

## 8. 비용 결과 저장 원칙

문서 기준으로 비용 계산은 MVP 필수 항목이다.
따라서 비용 결과 저장 구조에도 아래 규칙을 반영해야 한다.

- 모든 비용 응답에는 가정값을 포함한다.
- 리전, 인스턴스 타입, 사용시간 같은 기준값을 함께 저장한다.
- 총합과 상세 항목을 분리한다.
- 계산 실패 시 원인을 에러 메시지로 남긴다.
- 계산 복잡도가 높으면 단일값 대신 범위값으로 대체 가능하게 설계한다.

추가 확장 가능 필드 예시:

- `monthly_min_total`
- `monthly_max_total`
- `pricing_version`

MVP에서는 `monthly_total` 하나로 시작하고,
복잡해지면 범위형 필드를 추가하는 방식이 적절하다.

## 9. 이력 관리 초안

이력 관리는 Phase 2로 밀릴 수 있지만,
최소한의 기반은 처음부터 고려하는 것이 좋다.

권장 방식:

- 같은 프로젝트 안에서 세션마다 `version_no` 증가
- 각 세션은 입력, JSON, Terraform, 비용 결과를 한 세트로 가짐
- 이전 세션과 현재 세션을 비교하는 기능은 후순위

최소 구현 기준:

- 목록 조회 시 `version_no`, `status`, `created_at`이 보여야 한다.
- 세션 상세 조회가 가능해야 한다.

후속 확장:

- 이전/현재 JSON diff
- Terraform diff
- 비용 차이 비교

## 10. 추천 구현 순서

서비스 파이프라인 담당자는 아래 순서로 구현하는 것이 가장 안전하다.

### 1단계. 도메인 구조 확정

- `Project` 엔티티 정의
- `Session` 엔티티 정의
- 상태 enum 확정

### 2단계. 저장 구조 구현

- 세션 생성 로직 구현
- 프로젝트별 세션 목록 조회 구현
- 세션 상세 조회 구현

### 3단계. 결과 저장 API 구현

- FastAPI 아키텍처 JSON 저장 API
- FastAPI Terraform 결과 저장 API
- FastAPI 비용 결과 저장 API

### 4단계. 상태 흐름 연결

- 생성 파이프라인과 상태 변경 연결
- 실패 시 에러 저장 규칙 연결

### 5단계. 비용 계산 연결

- 비용 계산 모듈 구현 또는 연결
- 가정값 포함 응답 저장

### 6단계. 이력 최소 지원

- 버전 번호 증가 규칙 적용
- 세션 목록 조회로 버전 이력 확인 가능하게 구성

### 7단계. 후순위 확장

- 상세 비교 기능
- 이벤트 로그
- 재시도 정책 고도화

- 비용 범위형 고도화

## 12. 최종 정리

서비스 파이프라인 담당자의 첫 목표는
"생성 결과를 저장하고 다시 꺼내볼 수 있는 서비스 구조"를 만드는 것이다.

가장 먼저 확정해야 하는 것은 아래 3가지다.

- `Project`와 `Session`의 관계
- 세션 상태값과 상태 전이 규칙
- 결과 저장 및 조회 API 구조

이 3가지가 먼저 정리되어야 생성 파이프라인 결과와 프론트 UI가 자연스럽게 연결된다.
